### 1.1. Зачем нужна изоляция

Когда две сессии работают с одними и теми же строками **одновременно**, СУБД должна решить: какое чтение «правильное», можно ли видеть чужие незакоммиченные изменения, можно ли перезаписать чужое обновление. **Уровень изоляции** — это договорённость о том, какие аномалии допустимы ради скорости.

В этой работе важно различать два слоя:

1. **Аномалии по стандарту SQL** (dirty / non-repeatable / phantom read) — про то, *что* видит транзакция при повторных запросах.
2. **Lost update** — про то, *как* приложение записывает данные (шаблон read-modify-write). Её можно получить даже при «строгой» изоляции, если логика в коде неверная.

PostgreSQL по умолчанию использует **`READ COMMITTED`** и **MVCC**: каждый `SELECT` видит снимок данных на момент начала *этого* запроса, а не всей транзакции. Поэтому часть аномалий здесь **воспроизводится**, а dirty read — **нет**.

## 3. Dirty Read (грязное чтение)

### 3.1. Таблица и данные

Таблица `bank_accounts`. У записи `id = 1` (Алиса) баланс **5000.00**.

### 3.2. Две параллельные транзакции

| | T1 (`t1.sql`) | T2 (`t2.sql`) |
|---|---------------|---------------|
| Уровень | `READ COMMITTED` | `READ UNCOMMITTED` (в PostgreSQL = `READ COMMITTED`) |
| Действия | `UPDATE` −2000 → 3000, `pg_sleep(10)`, `ROLLBACK` | `SELECT balance` пока T1 открыта |

### 3.3. Шаги воспроизведения

1. `01_dirty_read/init.sql`
2. Сессия T1: `t1.sql`
3. Сессия T2: `t2.sql` (пока T1 не завершилась)

### 3.4. Результат

**Исходные данные:**

![Исходные данные bank_accounts](image/dr1.png)

**T1** — незакоммиченное значение 3000, после отката снова 5000:

![T1: update и rollback](image/dr3.png)

**T2** — чтение при открытой T1:

![T2: balance = 5000, не 3000](image/dr2.png)

**Вывод:** T2 видит **5000.00**, а не 3000.00. В PostgreSQL **грязное чтение не воспроизводится** (MVCC, уровень `READ UNCOMMITTED` фактически как `READ COMMITTED`).

### 3.4.1. Анализ

Сценарий моделирует классическую опасность: T1 списала деньги, T2 «увидела» новый баланс, а T1 откатилась — клиент принял решение по данным, которых в БД уже нет. В учебниках это и есть **dirty read**: чтение **незакоммичённой** версии строки.

В PostgreSQL `SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED` **не отключает** защиту: движок всё равно отдаёт только зафиксированные версии строк (MVCC). T2 в момент `SELECT` не видит «грязные» 3000, потому что снимок для запроса строится из committed-версий. Откат T1 для T2 «прозрачен» — она изначально читала стабильные 5000.

Это не значит, что dirty read «не существует» в принципе: в других СУБД (исторически MS SQL Server с `NOLOCK`, некоторые режимы MySQL) при слабом уровне изоляции его можно поймать. Здесь важен вывод: **поведение зависит от конкретной СУБД**, а не только от названия уровня в SQL-стандарте.

### 3.5. Как избежать

| Подход | Смысл |
|--------|--------|
| Уровень **`READ COMMITTED`** и выше | Не читать незакоммичённые изменения других транзакций |
| Не полагаться на `READ UNCOMMITTED` в PostgreSQL | В PG он не даёт «грязного» чтения — это ловушка для переноса скриптов из других СУБД |
| Короткие транзакции + явный `COMMIT`/`ROLLBACK` | Меньше окно, когда чужая незавершённая работа вообще могла бы мешать (в системах, где RC слабее) |

**На практике:** для финансовых и учётных операций в PostgreSQL достаточно дефолтного `READ COMMITTED`. Если переносите код из SQL Server/MySQL — перепроверьте хинты вроде `WITH (NOLOCK)` / `READ UNCOMMITTED`: в PostgreSQL они не эквивалентны «чтению грязи».

## 4. Non-repeatable Read (неповторяемое чтение)

### 4.1. Таблица и данные

Таблица `employees`. Сотрудник `id = 1` (Иванов), зарплата **50000.00**.

### 4.2. Две параллельные транзакции

| | T1 (`t1.sql`) | T2 (`t2.sql`) |
|---|---------------|---------------|
| Действия | два `SELECT salary`, между ними `pg_sleep(10)` | через 3 с: `UPDATE salary = 65000`, `COMMIT` |

### 4.3. Шаги воспроизведения

1. `02_non_repeatable_read/init.sql`
2. Параллельно `t1.sql` и `t2.sql`

### 4.4. Результат

**Подготовка:**

![init employees](image/nr4.png)

**T2** — обновление и commit:

![T2: salary 65000](image/nr2.png)

**T1** — второе чтение дало **другое** значение:

![T1: second read 65000](image/nr3.png)

**Вывод:** в одной транзакции T1 первое чтение — **50000**, второе — **65000**. Аномалия **воспроизведена** на уровне `READ COMMITTED`.

### 4.4.1. Анализ

T1 дважды читает **одну и ту же строку** (`WHERE id = 1`), ожидая стабильный результат — например, расчёт премии от зарплаты. Между чтениями T2 меняет зарплату и делает `COMMIT`. На `READ COMMITTED` в PostgreSQL **каждый** `SELECT` получает свежий снимок уже зафиксированных данных, поэтому второе чтение закономерно показывает **65000**.

Это не ошибка приложения и не «баг PostgreSQL» — так устроен уровень по умолчанию: баланс между согласованностью *внутри одной транзакции* и параллелизмом. Аномалия проявляется, когда бизнес-логика **неявно предполагает**, что «прочитал один раз — значит, до конца транзакции значение не изменится».

### 4.5. Как избежать

**1. Поднять уровень изоляции для всей транзакции:**

```sql
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- оба SELECT увидят один и тот же snapshot на момент первого запроса в транзакции
SELECT salary FROM employees WHERE id = 1;
-- ... T2 может закоммитить изменение, но T1 при втором SELECT всё ещё увидит 50000
SELECT salary FROM employees WHERE id = 1;
COMMIT;
```

**2. Зафиксировать значение один раз** — прочитали в переменную (в приложении или `SELECT … INTO`) и дальше работаете с ней, не перечитывая без необходимости.

**3. Пессимистичная блокировка**, если нужно запретить чужие изменения до конца своей транзакции:

```sql
SELECT salary FROM employees WHERE id = 1 FOR UPDATE;
```

**4. `SERIALIZABLE`** — если в транзакции много связанных чтений/записей и нужна максимальная согласованность (цена — больше откатов `serialization_failure`, нужен retry в коде).

**Выбор:** для отчётов «прочитал дважды — должно совпасть» обычно хватает **`REPEATABLE READ`**; `SERIALIZABLE` — когда критична сериализуемость всего сценария целиком.

## 5. Phantom Read (фантомное чтение)

### 5.1. Таблица и данные

Таблица `products`. Запрос: `COUNT(*)` WHERE `price > 30000` — изначально **3** строки.

### 5.2. Две параллельные транзакции

| | T1 (`t1.sql`) | T2 (`t2.sql`) |
|---|---------------|---------------|
| Действия | два одинаковых `COUNT(*)`, `pg_sleep(10)` | `INSERT` «Видеокарта» 60000, `COMMIT` |

### 5.3. Шаги воспроизведения

1. `03_phantom_read/init.sql`
2. Параллельно `t1.sql` и `t2.sql`

### 5.4. Результат

**Подготовка:**

![init products](image/pr1.png)

**T1** — второй подсчёт **4** (было 3):

![T1: second count 4](image/pr2.png)

**T2** — вставка строки в тот же диапазон:

![T2: count 4 after insert](image/pr3.png)

**Вывод:** во втором range-query появилась «фантомная» строка — count **3 → 4**. Аномалия **воспроизведена**.

### 5.4.1. Анализ

Отличие от non-repeatable read: меняется не только **существующая** строка, а **набор строк**, попадающих в условие (`price > 30000`). T1 «думает», что дорогих товаров три, а после вставки T2 во втором `COUNT` появляется четвёртая — **фантом**. Для бизнеса это та же проблема, что и с зарплатой: решение принято по устаревшему объёму выборки (лимит склада, налоговая база, агрегат в отчёте).

На `READ COMMITTED` каждый `COUNT(*)` снова смотрит на актуальные committed-данные. В PostgreSQL на **`REPEATABLE READ`** snapshot фиксируется на всю транзакцию: второй `COUNT` в T1 **не увидел бы** вставку T2 (фантом исчезает для читателя). В MySQL/InnoDB `REPEATABLE READ` часто закрывает и фантомы за счёт gap-lock — в PostgreSQL для жёсткой сериализации всего сценария иногда нужен **`SERIALIZABLE`**.

### 5.5. Как избежать

**1. `REPEATABLE READ`** — для повторяющихся range-запросов в одной транзакции:

```sql
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT COUNT(*) FROM products WHERE price > 30000;  -- 3
-- T2: INSERT ... COMMIT — для T1 «невидим»
SELECT COUNT(*) FROM products WHERE price > 30000;  -- снова 3
COMMIT;
```

**2. Блокировка диапазона** (если нужно не только читать стабильно, но и запретить вставки конкурентам):

```sql
SELECT COUNT(*) FROM products WHERE price > 30000 FOR UPDATE;
-- или в PostgreSQL для явной блокировки таблицы в редких сценариях: LOCK TABLE products IN SHARE ROW EXCLUSIVE MODE;
```

**3. Материализовать критерий** — сохранить список `id` или снимок отчёта в временную таблицу в начале транзакции и считать по ней.

**4. `SERIALIZABLE`** — когда несколько транзакций одновременно вставляют/удаляют в пересекающихся диапазонах и нужна полная сериализуемость (с обработкой откатов в приложении).

**На практике:** отчёты «срез на момент времени» → `REPEATABLE READ` или один снимок (реплика, `pg_export_snapshot`); жёсткие инвентаризации → блокировки или `SERIALIZABLE`.

## 6. Lost Update (потерянное обновление)

### 6.1. Таблица и данные

Таблица `warehouse`. Товар `id = 1`, **quantity = 10**.

### 6.2. Две параллельные транзакции (проблемный сценарий)

| | T1 (`t1_lost.sql`) | T2 (`t2_lost.sql`) |
|---|-------------------|-------------------|
| Действия | читает 10, пишет `quantity = 7` (10−3) | читает 10, пишет `quantity = 5` (10−5) |

Ожидаемый результат при корректной последовательности: **2** (10−3−5).  
Фактический — **7** (обновление T2 потеряно).

### 6.3. Шаги воспроизведения

1. `04_lost_update/init.sql`
2. Параллельно `t1_lost.sql` и `t2_lost.sql`

### 6.4. Результат (аномалия)

**T1** — чтение исходного значения:

![T1 reads quantity 10](image/lu2.png)

**T1** — итог в БД после обеих транзакций:

![T1 final check quantity 7](image/lu3.png)

**Вывод:** обе транзакции читали **10**, но T1 записала **7** поверх изменения T2. Одно списание **потеряно** — классический **lost update** при шаблоне read-modify-write без блокировки.

### 6.4.1. Анализ

Обе транзакции выполняют один и тот же антипаттерн:

1. `SELECT quantity` → получили **10**
2. в приложении/скрипте посчитали новое значение (7 и 5)
3. `UPDATE warehouse SET quantity = …` — **перезаписали** колонку целиком

T2 успела закоммитить **5**, но T1, опираясь на устаревшее «10−3», записала **7** и затерла результат T2. Ожидали **2** (10−3−5), получили **7**. Это не «чтение фантома», а **гонка записи**: уровень `REPEATABLE READ` сам по себе не спасёт, если обе транзакции читают одно и то же старое значение и пишут абсолютное число.

Правило: **lost update лечится не только изоляцией, а правильной моделью обновления** — блокировкой, проверкой версии или атомарным выражением в одном `UPDATE`.

### 6.5. Как избежать (продемонстрировано)

#### Способ 1: пессимистичная блокировка `FOR UPDATE`

Скрипты: `t1_for_update.sql`, `t2_for_update.sql`.

**T1** — блокировка строки:

![T1 FOR UPDATE lock](image/lu1.png)

**T2** — ожидание lock и корректный финал **2** (7−5):

![T2 final check quantity 2](image/lu8.png)

Первая транзакция держит **row-level lock** до `COMMIT`; вторая ждёт на `SELECT … FOR UPDATE`, затем читает уже **7** и пишет **2**. Порядок операций становится линейным — обновления не затирают друг друга. Минус: при высокой конкуренции за одну строку растут задержки и риск **deadlock** (нужен единый порядок блокировок по `id`).

#### Способ 2: optimistic locking (версия строки)

Скрипты: `optimistic_prepare.sql`, `t1_optimistic.sql`, `t2_optimistic.sql`.

**Подготовка** — колонка `version`:

![optimistic prepare](image/lu9.png)

**T2** — успешное обновление (`UPDATE 1`):

![T2 optimistic success](image/lu10.png)

**T1** — версия уже изменена (конфликт; далее `UPDATE 0`):

![T1 reads version 2 after T2](image/lu.png)

**Вывод:** T2 успела закоммитить с `version = 2`; T1 при `UPDATE … WHERE version = 1` не обновляет строку (`UPDATE 0`) — **lost update предотвращён**. В реальном API после `UPDATE 0` нужно **перечитать строку и повторить** операцию (retry), иначе пользователь не узнает, что списание не прошло.

#### Способ 3: атомарный UPDATE

```sql
UPDATE warehouse SET quantity = quantity - 3 WHERE id = 1;
```

без предварительного чтения в приложении — СУБД сама читает актуальную строку под блокировкой и применяет инкремент. **Предпочтительный вариант** для счётчиков остатков, лайков, балансов.

#### Когда что выбирать

| Способ | Плюсы | Минусы |
|--------|--------|--------|
| Атомарный `UPDATE … = col ± N` | Просто, быстро, мало кода | Не подходит, если формула сложная и считается только в приложении |
| `FOR UPDATE` | Гарантированная последовательность | Блокировки, риск deadlock, ниже параллелизм |
| Optimistic (`version`) | Высокая параллельность при редких конфликтах | Нужен retry при `UPDATE 0`, колонка версии в схеме |
| `REPEATABLE READ` + тот же read-modify-write | — | **Не устраняет** lost update при записи абсолютного значения |

В ORM опасны цепочки `obj.quantity -= 3; session.commit()` без блокировки — под капотом часто именно read-modify-write.

## 8. Общие выводы

1. **PostgreSQL на `READ COMMITTED` (по умолчанию)** защищает от dirty read, но **допускает** non-repeatable и phantom read — мы это увидели на `employees` и `products`.

2. **Lost update** — отдельный класс проблем: он возникает из **логики записи**, а не только из уровня изоляции. Даже `SERIALIZABLE` не заменит атомарный `UPDATE`, если приложение перезаписывает поле целиком по устаревшему чтению.

3. **Практическая стратегия:**
   - обычные CRUD и короткие транзакции → оставить `READ COMMITTED`;
   - «прочитал дважды — должно совпасть» / отчётный срез → `REPEATABLE READ` или один снимок;
   - остатки, балансы, счётчики → `UPDATE … SET col = col ± ?` или `FOR UPDATE` / `version`;
   - при конфликтах optimistic locking → повтор транзакции в коде.

4. **Перед переносом SQL между СУБД** сверять документацию: `READ UNCOMMITTED`, `REPEATABLE READ` и блокировки ведут себя по-разному; учебный стандарт SQL описывает аномалии, а конкретный движок решает, какие из них реально возможны.
