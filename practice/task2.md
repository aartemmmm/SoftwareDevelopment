## 2. Схема потока сообщений

### Обычный путь

```

1. Producer → отправляет сообщение (message_id)
2. Broker → кладёт в main topic
3. Worker → читает сообщение
4. Worker → проверяет message_id в БД
5. Если нет:
   → обрабатывает
   → пишет результат в Audit DB
   → сохраняет message_id
6. Worker → отправляет ACK в Broker

```

---

### Путь ошибки

```

1. Worker получает сообщение
2. Ошибка обработки
3. Retry (N раз)
4. Если превышен лимит:
   → сообщение отправляется в DLQ

```

---

## 3. Exactly-once

Реализация через **idempotency + транзакцию**:

- у каждого сообщения есть `message_id`
- таблица:
```

processed_messages(message_id PRIMARY KEY)

```

Алгоритм:
```

1. Проверка message_id
2. Если нет:

   * запись результата в Audit
   * запись message_id
     (в одной транзакции)
3. После успешной записи → ACK

```

→ дубликаты не обрабатываются повторно

---

## 4. DLQ

- используется после N неудачных попыток (retry)
- сообщение отправляется в отдельную очередь DLQ

Содержит:
- message_id
- payload
- ошибка
- retry count

Дальше:
- ручная обработка или отдельный consumer
